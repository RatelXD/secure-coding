# 06. 보안 약점과 개선 계획

## 6.1 작성 원칙

이 장은 Tiny Second-hand Shopping Platform의 실제 기능과 직접 연결되는 위험만 다룹니다. GitHub 리뷰 방식, 배포 승인 절차와 같은 저장소 운영 기록은 과제 본문에서 제외했습니다.

현재 저장소에는 설계와 구현 중 발견한 위험에 대한 방어와 음성·회귀 테스트가 반영되어 있습니다. 자동 검증이 끝난 항목과 실제 배포·운영 환경에서 추가 확인할 항목을 구분해 기록합니다. 각 표의 ‘발견 당시 상태’는 수정 전의 역사적 기록이며 현재 상태를 뜻하지 않습니다.

## VULN-01. 로그인 무차별 대입과 계정 정보 노출

|---|---|
| 식별자 | `VULN-01` |

| 위험 | 공격자가 반복 로그인으로 비밀번호를 추측하거나 응답 차이를 이용해 존재하는 아이디를 찾을 수 있습니다. |
| 영향 | 사용자 계정과 세션 |

| 개선 조치 | Django 비밀번호 해시·검증기를 사용하고, 계정과 비식별 HMAC IP 키 기준으로 실패 횟수와 냉각 시간을 적용했습니다. IP 제한을 먼저 확인하고 24시간 지난 상태를 정리하며, 누락·변조·불일치 세션 epoch는 로그아웃합니다. 성공 시 계정 실패 기록만 초기화하고 존재 여부와 관계없이 같은 오류를 반환합니다. |
| 검증 | 5회·20회 경계, 동일 계정·IP 병렬 실패, 차단 뒤 임의 아이디 저장 증폭, 알 수 없는 아이디, 제한 만료와 세션 epoch 음성 테스트를 실행했습니다. |
| 잔여 위험 | 여러 계정·여러 네트워크를 이용한 저속 공격은 완전히 막을 수 없어 패턴 관찰이 필요합니다. |

| 항목 | 내용 |
|---|---|
| 식별자 | `VULN-02` |
| 발견 단계 | 요구사항·설계·구현 |
| 위험 | 요청의 사용자·소유자 ID나 임의 필드를 그대로 신뢰하면 다른 사용자의 프로필·상품·신고를 변경하거나 권한 필드를 바꿀 수 있습니다. |

| 개선 조치 | 행위자는 세션에서 결정하고, 프로필은 본인 세션만 사용하며 상품은 조회와 행 잠금에서 소유자를 다시 확인합니다. 입력은 Form allowlist만 받고, 존재를 숨겨야 하는 타인 상품과 채팅방은 404로 처리합니다. |
| 검증 | 타인 상품 수정·삭제, 임의 권한 필드, 제3자 1대1 채팅 접근, 비인증 보호 URL과 CSRF 없는 변경 요청을 음성 테스트로 확인했습니다. |
| 잔여 위험 | 새 URL·WebSocket을 추가할 때 같은 정책을 누락할 수 있어 진입점별 회귀 테스트가 필요합니다. |

## VULN-04. 채팅 권한 우회·XSS·중복 메시지

| 항목 | 내용 |
|---|---|
| 식별자 | `VULN-04` |
| 발견 단계 | 설계·구현 |
| 위험 | 비인증 연결, 위조 Origin, 타인 방 접근, HTML 실행, 과도한 전송, ACK 유실 뒤 재전송이 권한 침해·XSS·중복 저장으로 이어질 수 있습니다. |
| 영향 | 채팅 참여자, 채팅 이력, 서비스 가용성 |
| 발견 당시 상태 | 방·참여자·메시지 모델과 client UUID 제약만 있었고 consumer가 없었습니다. 초기 구현에서는 nullable join에 방 잠금을 적용해 PostgreSQL 오류가 발생했고, Host userinfo와 휴면 뒤 기존 소켓 수신을 차단하지 못했습니다. |
| 개선 조치 | 연결에서 인증·정확 Origin을 확인하고, frame과 outbound 전달마다 참여자·현재 계정 상태를 다시 확인합니다. 방 행만 잠그고 DB에 `degraded`를 먼저 커밋한 뒤 전달 성공 시 `live`로 전환합니다. 본문은 text로 출력하고 UUIDv4 재전송은 중복 저장·재발행하지 않습니다. |
| 검증 | 비인증·누락/위조 Origin·Host userinfo·제3자 방·2000/2001바이트·속도 경계·Redis 장애·재전송 충돌·휴면 뒤 수신 차단을 자동 테스트로 확인했습니다. |
| 잔여 위험 | Redis 장애 동안 실시간 전달은 중단될 수 있으며, 이력 동기화가 누락되면 사용자 화면에 일시적인 차이가 남습니다. |

## VULN-07. 신뢰되지 않은 프록시 헤더

| 항목 | 내용 |
|---|---|
| 식별자 | `VULN-07` |
| 발견 단계 | 설계·구현·테스트 |
| 위험 | 애플리케이션이 모든 `X-Forwarded-*` 값을 신뢰하면 외부 사용자가 HTTPS 여부나 Host 정보를 위조해 리다이렉트·CSRF·출처 판단을 우회할 수 있습니다. |
| 영향 | HTTPS 강제, Host 검증, CSRF 출처와 보안 쿠키 판단 |
| 발견 당시 상태 | 프록시 헤더 신뢰를 켜면 전달된 Host까지 사용했으며 실제 프록시 주소를 제한하는 설정이 없었습니다. |
| 개선 조치 | 전달된 Host는 사용하지 않고, 프록시 헤더를 켤 때 IP 리터럴 allowlist를 필수로 요구합니다. 첫 미들웨어는 허용된 프록시의 정확한 단일 `https` 주장만 받아들이고 나머지 전달 헤더를 제거하며, 운영 CSRF 출처는 명시적 HTTPS 값만 허용합니다. |
| 검증 | `tests/security/config/test_proxy_headers.py`의 정상·비허용 IP·다중 프로토콜·잘못된 IP 테스트와 `check --deploy`를 실행했습니다. |
| 잔여 위험 | 실제 배포 프록시가 클라이언트 전달 헤더를 덮어쓰는지와 TLS 종단 IP가 allowlist와 일치하는지는 배포 환경에서 다시 확인해야 합니다. |

## VULN-08. 운영 데이터베이스 TLS 설정 하향

| 항목 | 내용 |
|---|---|
| 식별자 | `VULN-08` |
| 발견 단계 | 구현·검토 |
| 위험 | 운영 설정이 PostgreSQL TLS 모드를 `disable`, `allow`, `prefer`로 받아들이면 데이터베이스 인증정보와 서비스 데이터가 평문 연결로 전송될 수 있습니다. |
| 영향 | 사용자·상품·채팅·신고 등 PostgreSQL에 저장되는 모든 데이터 |
| 발견 당시 상태 | 환경 변수의 값을 검증하지 않고 libpq `sslmode`에 그대로 전달했습니다. |
| 개선 조치 | 알려진 libpq 모드만 허용하고, 운영 환경에서는 최소 `require` 이상인 `require`, `verify-ca`, `verify-full`만 허용합니다. 개발·테스트 환경의 명시적 `disable`은 로컬 격리 실행에만 남겼습니다. |
| 검증 | 세 가지 하향 모드를 각각 운영 설정에 넣었을 때 시작이 거부되는지 자동 테스트하고, 허용된 TLS 설정으로 `check --deploy`를 실행했습니다. |
| 잔여 위험 | `require`는 암호화하지만 서버 신원까지 검증하지 않으므로 실제 운영에서는 CA와 호스트명을 검증하는 `verify-full`을 우선 적용해야 합니다. |
