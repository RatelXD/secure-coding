# 06. 보안 약점과 개선 계획

## 6.1 작성 원칙

이 장에서는 Tiny Second-hand Shopping Platform의 실제 기능과 맞닿은 위험, 방어 조치, 검증 상태만 다룹니다.

저장소에는 설계와 구현 과정에서 확인한 위험에 대한 방어 조치와 음성·회귀 테스트를 반영해 두었습니다. 자동 검증을 마친 항목과 실제 배포·운영 환경에서 추가로 확인할 항목은 구분해 기록합니다. 각 표의 ‘발견 당시 상태’는 수정 전의 기록일 뿐 현재 상태를 뜻하지 않습니다.

## VULN-04. 채팅 권한 우회·XSS·중복 메시지

| 항목 | 내용 |
|---|---|
| 식별자 | `VULN-04` |
| 발견 단계 | 설계·구현 |

| 영향 | 채팅 참여자, 채팅 이력, 서비스 가용성 |
| 발견 당시 상태 | 방·참여자·메시지 모델과 client UUID 제약만 있었고 consumer가 없었습니다. 초기 구현에서는 nullable join에 방 잠금을 적용해 PostgreSQL 오류가 발생했고, Host userinfo와 휴면 뒤 기존 소켓 수신을 차단하지 못했습니다. |
| 개선 조치 | 연결 시 인증과 정확한 Origin을 확인하고, frame과 outbound 전달 때마다 참여자와 현재 계정 상태를 다시 확인합니다. 방 행만 잠그고 DB에 `degraded`를 먼저 커밋한 뒤 전달에 성공하면 `live`로 전환합니다. 본문은 text로 출력하며 UUIDv4 재전송은 중복 저장하거나 재발행하지 않습니다. |
| 검증 | 비인증·누락/위조 Origin·Host userinfo·제3자 방·2000/2001바이트·속도 경계·Redis 장애·재전송 충돌·휴면 뒤 수신 차단을 자동 테스트로 확인했습니다. |
| 잔여 위험 | Redis 장애 동안 실시간 전달은 중단될 수 있으며, 이력 동기화가 누락되면 사용자 화면에 일시적인 차이가 남습니다. |

PR #43·#44에서 연결 성공 상태를 `상품 대화 · 연결됨`으로 명시하고, 4400·4403·1000 종료는 재시도하지 않도록 했습니다. 그 밖의 비정상 종료는 최대 5회 지수 백오프로 제한하며 10초 안정 연결 뒤에만 재시도 횟수를 초기화합니다. 실제 배포 프록시가 WebSocket upgrade와 close code를 보존하는지는 여전히 배포 환경에서 확인해야 합니다.

## VULN-08. 운영 데이터베이스 TLS 설정 하향

| 항목 | 내용 |
|---|---|
| 식별자 | `VULN-08` |
| 발견 단계 | 구현·검토 |
| 위험 | 운영 설정이 PostgreSQL TLS 모드를 `disable`, `allow`, `prefer`로 받아들이면 데이터베이스 인증정보와 서비스 데이터가 평문 연결로 전송될 수 있습니다. |
| 영향 | 사용자·상품·채팅·신고 등 PostgreSQL에 저장되는 모든 데이터 |
| 발견 당시 상태 | 환경 변수의 값을 검증하지 않고 libpq `sslmode`에 그대로 전달했습니다. |
| 개선 조치 | 알려진 libpq 모드만 허용합니다. 운영 환경에서 허용하는 최소 수준은 `require`이며, 허용 목록은 `require`, `verify-ca`, `verify-full`뿐입니다. 개발·테스트 환경에서 명시한 `disable`은 로컬 격리 실행에만 남겼습니다. |
| 검증 | 세 가지 하향 모드를 각각 운영 설정에 넣었을 때 시작이 거부되는지 자동 테스트하고, 허용된 TLS 설정으로 `check --deploy`를 실행했습니다. |
| 잔여 위험 | `require`는 암호화하지만 서버 신원까지 검증하지 않으므로 실제 운영에서는 CA와 호스트명을 검증하는 `verify-full`을 우선 적용해야 합니다. |

## 6.3 2차 위협 모델과 현재 통제

아래 세 항목은 2차 기능을 구현하기 전에 작성한 위협 분석을 현재 코드 기준으로 갱신한 것입니다. 제품 코드는 main에 반영됐지만, 모든 세부 경계·장애 시나리오를 같은 조건으로 재실행했다는 뜻은 아니므로 상태와 잔여 검증을 분리합니다.

## VULN-09. 모의 이체의 중복·부분 처리와 잔액 불일치

| 항목 | 내용 |
|---|---|
| 식별자 | `VULN-09` |
| 발견 단계 | 2차 요구사항·설계 |
| 위험 | 재전송, 같은 멱등성 키의 변조, 잔액 확인과 차감 사이의 경합, 반대 방향 교착, 처리 중 종료가 중복 이체·음수 잔액·한쪽만 반영된 원장을 만들 수 있습니다. |
| 영향 | 모의 계정 잔액, 이체 결과, 감사와 복구 가능성 |
| 구현 상태 | 이체 모델·마이그레이션·서비스·화면이 구현되어 있습니다. 화면·HTTP 입력은 1~99,999,999 정수 원화이고 내부 원장·canonical payload는 `Decimal(12,2)` 둘째 자리를 유지합니다. |
| 개선 조치 | 세션 사용자를 송신자로 고정하고 양쪽 활성·미종료 상태를 잠금 안에서 재검사합니다. 초기 `SEED_ISSUE`는 사용자 `+100,000.00`과 비공개 `SEED_RESERVE` `-100,000.00`의 합계 0 journal이며 계정당 한 번만 발행합니다. |
| 동시성·불변성 | safety shared lock 뒤 멱등 lock으로 저장 결과를 먼저 조회하고 새 키만 OPEN을 확인한 뒤 계정 PK lock을 얻습니다. 따라서 BLOCKED 중 저장된 같은-payload 201·422는 재현하지만 새 이체는 막습니다. deferred trigger는 모든 journal의 정확히 두 entry·합계 0을 검증하고 journal/entry UPDATE·DELETE trigger가 변경을 막습니다. |
| 실패·복구 | 대사·재개는 exclusive safety lock으로 이체·종료와 직렬화합니다. 불일치는 incident와 함께 시스템을 BLOCKED로 만들며 maintainer만 전체 불일치 0건을 확인해 재개합니다. |
| 구현 뒤 필수 검증 | 비정상 journal 커밋, journal/entry 변경, canonical/병렬 이체, BLOCKED barrier 경합, BLOCKED 중 저장 201·422 재현과 새 키 503, 총 4회 재시도, 무권한·불완전·정상 복구를 확인합니다. |
| 잔여 위험 | DB 운영자 권한 탈취와 애플리케이션 밖의 직접 변경은 애플리케이션 통제만으로 완전히 막을 수 없어 최소 DB 권한, 백업·복원, 정기 reconciliation이 필요합니다. |
| 계정 수명 | `SEED_ISSUE` 100,000.00은 계정당 한 번만 발행합니다. 인증된 소유자가 잔액 0에서 계정 행 잠금으로 닫고 원장·감사는 보존하며, 닫힌 계정의 송수신은 거부합니다. |
| 상태 | 구현됨·PR #43의 unit/integration/security CI PASS, 상세 경계·장애 재검증 필요 |
