# 06. 보안 약점과 개선 계획

## 6.1 작성 원칙

이 장은 Tiny Second-hand Shopping Platform의 실제 기능과 직접 연결되는 위험만 다룹니다. GitHub 리뷰 방식, 배포 승인 절차와 같은 저장소 운영 기록은 과제 본문에서 제외했습니다.

현재 제품 코드는 공개 `main` 통합 전이므로 설계 단계에서 발견한 위험과 일부 코드 골격만 기록합니다. 음성 테스트와 회귀 테스트가 끝나지 않은 항목은 `개선 완료`로 표현하지 않습니다.

## VULN-02. 타인 객체 접근과 대량 할당

| 항목 | 내용 |
|---|---|
| 식별자 | `VULN-02` |
| 발견 단계 | 요구사항·설계·구현 |
| 위험 | 요청의 사용자·소유자 ID나 임의 필드를 그대로 신뢰하면 다른 사용자의 프로필·상품·신고를 변경하거나 권한 필드를 바꿀 수 있습니다. |
| 영향 | 프로필, 상품, 신고, 관리자 기능 |
| 기존 상태 | 사용자·상품 서비스 골격에 변경 가능 필드와 변경 금지 필드가 분리되어 있지만 실제 HTTP 권한 검사는 아직 없습니다. |
| 개선 조치 | 행위자는 세션에서 결정하고, 객체별 소유권을 서버에서 조회합니다. 입력은 allowlist 필드만 받아 모델 전체 갱신을 피합니다. 존재를 숨겨야 하는 객체는 권한 실패 시 404를 사용합니다. |
| 검증 | 일반 사용자가 타인 프로필·상품·신고 ID와 `is_staff`, `owner_id`, `auth_epoch` 등을 보낸 음성 테스트가 필요합니다. |
| 잔여 위험 | 새 URL·WebSocket·관리 작업을 추가할 때 같은 정책을 누락할 수 있어 진입점별 회귀 테스트가 필요합니다. |

| 잔여 위험 | 이미지 라이브러리 자체 취약점과 CPU·메모리 고갈 위험은 의존성 갱신과 자원 제한으로 계속 관리해야 합니다. |

## VULN-04. 채팅 권한 우회·XSS·중복 메시지

| 항목 | 내용 |
|---|---|
| 식별자 | `VULN-04` |
| 발견 단계 | 설계·구현 |
| 위험 | 비인증 연결, 위조 Origin, 타인 방 접근, HTML 실행, 과도한 전송, ACK 유실 뒤 재전송이 권한 침해·XSS·중복 저장으로 이어질 수 있습니다. |
| 영향 | 채팅 참여자, 채팅 이력, 서비스 가용성 |
| 기존 상태 | 방·참여자·메시지 모델과 client UUID 고유 제약, 저장 우선 서비스 계약은 있으나 consumer와 종단 검증은 아직 없습니다. |
| 개선 조치 | handshake에서 인증·허용 Origin을 확인하고, frame 처리 시 참여자·계정 상태·길이·속도를 다시 확인합니다. 본문은 text로 저장하고 화면에서 escape합니다. DB 커밋 뒤에만 수락 응답을 보내며 같은 UUID 재전송은 기존 결과를 반환합니다. |
| 검증 | 비인증·null/위조 Origin·제3자 방·2000/2001바이트·속도 경계·Redis 장애·ACK 유실 테스트가 필요합니다. |
| 잔여 위험 | Redis 장애 동안 실시간 전달은 중단될 수 있으며, 이력 동기화가 누락되면 사용자 화면에 일시적인 차이가 남습니다. |

## VULN-05. 신고 조작과 동시 제재 경합

| 항목 | 내용 |
|---|---|
| 식별자 | `VULN-05` |
| 발견 단계 | 요구사항·설계·구현 |
| 위험 | 자기·중복·신규 계정 신고를 그대로 세거나 동시 요청을 분리 처리하면 공격자가 상품을 숨기고 사용자를 휴면시키거나 중복 제재를 만들 수 있습니다. |
| 영향 | 상품 노출, 사용자 접근, 감사 기록 |
| 기존 상태 | 신고자-대상 고유 제약, 대상 무결성, 시작·만료 시각 모델은 있으나 유효 신고 판정과 원자적 제재 서비스는 아직 없습니다. |
| 개선 조치 | 인증·활성·가입 7일·자기 신고 제외·최근 7일·독립 신고자 조건을 트랜잭션 안에서 확인합니다. 신고 소비, 임계 판정, 단일 제재, 감사 기록을 한 번에 저장하고 만료는 DB 현재 시각으로 계산합니다. |
| 검증 | 4·5·6명 경계, 사용자 맥락 1·2개, 동시 신고, 기존 소비 신고 재사용, 만료 직전·직후 테스트가 필요합니다. |
| 잔여 위험 | 다수 계정을 이용한 공모를 완전히 식별할 수 없으므로 속도 제한과 관리자 검토가 필요합니다. |

## VULN-06. 모의 이체의 중복·잔액 불일치

| 항목 | 내용 |
|---|---|
| 식별자 | `VULN-06` |
| 발견 단계 | 요구사항·설계 |
| 위험 | 동시에 이체하거나 응답을 받지 못해 재시도할 때 중복 차감, 음수 잔액, 부분 저장이 발생할 수 있습니다. |
| 영향 | 모의 잔액과 거래·감사 기록 |
| 기존 상태 | 모의 이체는 2차 기능으로 아직 코드가 없습니다. |
| 개선 조치 | 계정을 고정 순서로 잠그고 하나의 PostgreSQL 트랜잭션에서 잔액, 이체, 차변·대변 원장, 감사를 저장합니다. 사용자별 멱등성 UUID와 요청 payload hash를 보존합니다. |
| 검증 | 자기 이체, 0·음수·상한 초과, 잔액 부족, 동일·다른 payload 재전송, 동시 이체, 중간 실패, 차변·대변 합계 테스트가 필요합니다. |
| 잔여 위험 | 교착·직렬화 실패 재시도와 운영 복구 절차는 실제 PostgreSQL 환경에서 측정해야 합니다. 실제 결제 안전성을 보장하는 기능이 아닙니다. |

## VULN-07. 오류·로그의 민감정보 노출

| 항목 | 내용 |
|---|---|
| 식별자 | `VULN-07` |
| 발견 단계 | 요구사항·설계 |
| 위험 | 상세 예외, 비밀번호, 세션, 토큰, 내부 경로, 원문 채팅을 응답이나 로그에 남기면 추가 공격과 개인정보 노출에 이용될 수 있습니다. |
| 영향 | 모든 사용자와 운영 정보 |
| 기존 상태 | 공개 보고서는 비밀값과 개인정보를 제외하고 있으나 제품 오류·로그 경로는 아직 종합 검증하지 않았습니다. |
| 개선 조치 | 사용자 응답은 일반화하고 서버 로그에는 상관 ID와 사건 종류만 기록합니다. 비밀번호·세션·토큰·원문 채팅·원문 IP는 기록하지 않거나 비식별화합니다. 운영 환경에서는 디버그를 끕니다. |
| 검증 | 400·403·404·429·500 응답, 인증 실패, 업로드 실패, WebSocket 거부, 의도적 내부 예외의 화면·로그를 점검해야 합니다. |
| 잔여 위험 | 새 라이브러리와 예외 처리 코드가 민감값을 자동 출력할 수 있어 정기적인 로그 표본 점검이 필요합니다. |
