# 주요 기술 의사결정 요약

이 문서는 구현·검증된 1차 설계와 구현 전 작성한 2차 설계를 구분합니다. 2차 결정은 제품 코드가 존재한다는 뜻이 아니며 모두 `설계 완료·구현 예정` 상태입니다.

## DEC-01. 같은 출처의 단일 Django ASGI 애플리케이션

- **이유:** 세션, CSRF, Origin, 객체 권한을 한 애플리케이션에서 일관되게 처리하기 위해서입니다.
- **결정:** Django 5.2 LTS, Templates, vanilla JavaScript, Channels를 사용합니다.
- **데이터 경계:** PostgreSQL은 영속 데이터의 기준이고 Redis는 실시간 전달에만 사용합니다.
- **상태:** 구현·자동 검증 PASS.

## DEC-02. 채팅은 DB 저장 성공을 수락 기준으로 사용

- **문제:** Redis 전달 성공을 먼저 반환하면 DB 저장 실패 시 화면에만 보이는 메시지가 생길 수 있습니다.
- **결정:** client UUID로 중복을 식별하고 PostgreSQL 커밋 뒤에 수락 응답을 보냅니다.
- **장애 처리:** Redis 전달이 실패해도 저장은 유지하며, 클라이언트가 마지막 메시지 ID 이후 이력을 다시 조회합니다.
- **상태:** 구현·자동 검증 PASS.

## DEC-03. 제재 상태는 DB 현재 시각으로 계산

- **문제:** 별도 상태 플래그나 예약 작업에만 의존하면 만료 시점과 HTTP·WebSocket·조회 결과가 달라질 수 있습니다.
- **결정:** 시작·만료 시각과 DB 현재 시각으로 사용자 휴면과 상품 비노출 효력을 계산합니다.
- **제약:** 제재는 가역적이며 물리 삭제하지 않습니다.
- **상태:** 구현·자동 검증 PASS.

## DEC-04. 안전한 이미지 직접 업로드

- **이유:** 상품당 이미지 1장을 지원하면서 스크립트·메타데이터·과대 이미지 위험을 줄이기 위해서입니다.
- **결정:** JPEG·PNG·WebP, 5MiB, 4096×4096으로 제한하고 완전 디코딩 뒤 새 파일로 재인코딩합니다.
- **상태:** 구현·자동 검증 PASS.

## DEC-05. 모의 이체는 온라인 잔액과 불변 이중 분개를 함께 사용

- **문제:** 잔액만 저장하면 변경 근거를 증명하기 어렵고, 원장만 매번 합산하면 온라인 조회와 잠금 비용이 커집니다. 재전송·동시 요청·프로세스 종료는 중복 또는 부분 이체를 만들 수 있습니다.
- **결정:** `MockAccount.balance`를 온라인 권위값으로 사용합니다. 모든 journal은 정확히 두 entry와 합계 0이며 deferred constraint trigger가 커밋 시 검증합니다. journal/entry UPDATE·DELETE는 DB trigger로 거부하고 seed는 사용자 +100,000.00·`SEED_RESERVE` -100,000.00입니다. seed만 대상으로 하는 부분 고유 제약으로 계정당 한 번 발행하되 `TRANSFER`·`COMPENSATION`은 허용합니다.
- **이체 트랜잭션:** `Decimal(12,2)`의 1회 이체는 0.01~99,999,999.99, 결과 사용자 잔액은 0~1,000,000,000.00입니다. canonical payload는 버전, 변형하지 않은 수신자 아이디 원문, 금액 고정 2자리의 정렬 UTF-8 JSON입니다. stable digest `(sender,key)` advisory lock 뒤 저장 결과를 확인하고 두 계정을 PK 오름차순으로 잠급니다.
- **재시도 계약:** safety shared lock과 멱등성 lock 뒤 같은 키·payload의 최초 성공 또는 업무 거부 HTTP 상태·body를 `TransferSafetyState=BLOCKED` 중에도 영구 재현하고 다른 payload는 409로 거부합니다. OPEN 검사는 저장 결과가 없는 새 키에만 적용합니다. PostgreSQL `40001`·`40P01`만 최초 시도 뒤 새 트랜잭션으로 최대 3회 재시도하고 네 번 모두 실패하면 503으로 종료합니다.
- **금지:** Redis 원장, 실제 결제, 직접 잔액 수정, 원장 수정·삭제, 관리자 임의 조정은 허용하지 않습니다. 조정은 형식이 지정된 반대 분개로만 수행합니다.
- **계정 종료:** 인증된 소유자만 CSRF 보호 POST로 종료할 수 있습니다. 이체와 같은 계정 행 잠금에서 잔액 0·열린 상태를 재검사해 닫고, 원장·감사는 보존하며 이후 송수신을 거부합니다.
- **대안:** balance-only는 감사·복구가 약하고 ledger-only는 조회 비용과 구현 복잡도가 커서 선택하지 않았습니다. 별도 처리 중 카운터는 행 잠금과 충돌하고 stale 상태를 만들 수 있어 두지 않습니다.
- **결과와 복구:** 새 이체·종료는 safety shared lock 안에서 OPEN을 확인하고, 대사·재개는 exclusive lock을 사용해 BLOCKED 전환 뒤 늦은 새 커밋을 막습니다. 저장된 같은-payload 201·422 재현은 BLOCKED 중에도 유지합니다. maintainer는 전체 불일치 0건과 정확 incident를 확인해야 재개할 수 있습니다. 구현 뒤 barrier 경합과 무권한·불완전·정상 복구를 검증합니다.
- **상태:** 설계 완료·구현 예정.

## DEC-06. 검색은 공개 범위를 먼저 고정하고 결정적으로 페이지를 나눔

- **문제:** 검색 뒤에 공개 여부를 걸러내면 숨김 상품의 존재나 결과 수가 새고, 임의 정렬·큰 페이지·비결정적 동률 순서는 주입과 자원 고갈, 중복·누락을 일으킬 수 있습니다.
- **결정:** 기존 상품을 NFC로 backfill하고 쓰기 서비스와 DB CHECK로 저장값 NFC를 강제합니다. `q`도 trim→NFC 뒤 0~100자로 제한합니다. 공개 범위·allowlist 정렬·`1..500` 페이지를 적용합니다.
- **오류·비용 계약:** 잘못된 입력과 501 이상은 상품 쿼리 전에 400입니다. 검색 서비스 블록은 middleware/auth/session을 제외하고 count 1회+slice 1회만 사용하며 최대 20행을 반환합니다. 내부 SQL·검색어 원문은 기록하지 않습니다.
- **대안:** 전체 집합 검색 뒤 애플리케이션 필터링과 임의 ORM 정렬 문자열은 정보 노출과 주입 위험 때문에 허용하지 않습니다. 현재 보고서가 요구하는 번호 페이지와 전체 건수를 유지하기 위해 cursor 방식 대신 제한된 OFFSET을 사용합니다.
- **결과와 검증:** 조합/분해 문자, 정규화 전후 길이, 비노출 제외, 주입, 동률, 500/501, 인접 페이지와 서비스 블록 query budget을 검증해야 합니다.
- **상태:** 설계 완료·구현 예정.

## DEC-07. 관리 기능은 작업 권한·객체 범위·가역 변경으로 제한

- **문제:** `is_staff` 하나나 Django 기본 CRUD만 사용하면 역할보다 넓은 열람·변경, CSRF, 오래된 화면의 덮어쓰기, 감사 삭제가 가능합니다.
- **결정:** 일반 작업은 staff·codename·대상 grant를 요구합니다. `AdminScopeGrant`는 USER/PRODUCT FK 중 정확히 하나와 활성 중복 제약을 갖습니다. `manage_admin_scope`만 콘텐츠 grant가 없는 meta-scope 예외이며 superuser와 직접 permission을 모두 요구합니다.
- **권한 수명:** scope 변경도 300초 재인증, 10~500자 사유, CSRF, staff `auth_epoch`/grant version을 요구합니다. 최초·복구 permission은 유효 최고관리자 0명일 때 `scope_bootstrap` 역할의 관리 명령만 부여하며, 전용 advisory lock 안에서 0명을 재검사하고 permission과 감사를 한 트랜잭션으로 커밋합니다.
- **허용 범위:** `view_report`, `apply_sanction`, `release_sanction`, `view_admin_audit`만 grant 안에서 허용합니다. 비인증 HTML은 302, staff/codename 부족은 403, 미존재·범위 밖 대상은 같은 404입니다. 상태 변경은 UTC 재인증 나이 300초 이하, NFC code point 10~500자 사유, CSRF와 최신 version을 요구합니다.
- **금지:** 비밀번호 hash·세션·비밀값·채팅 본문 열람, 잔액·원장·감사의 수정·삭제, 영구 삭제, break-glass 우회는 허용하지 않습니다.
- **대안:** staff-only와 범용 superuser CRUD는 최소 권한을 만족하지 않아 배제했습니다. 외부 object-permission 의존성 대신 작은 DB-backed grant를 사용해 대상과 취소 의미를 명시합니다.
- **결과와 검증:** 감사 INSERT 실패는 업무와 함께 롤백해 503·DB 감사 0건이며 기존 감사 변경은 DB가 거부합니다. 제재는 적용 경합 시 활성/감사 각 1건, 정확한 7일 만료에는 활성 0·해제 기록 0, 만료 뒤 해제에는 409·충돌 감사 1건·`SanctionRelease` 0건, 조기 해제 경합에는 `SanctionRelease`/감사 각 1건입니다. 역할·범위·시간·사유·bootstrap 경합·감사 장애와 기록 수를 검증합니다.
- **상태:** 설계 완료·구현 예정.
